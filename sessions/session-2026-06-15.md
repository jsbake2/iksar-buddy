## INCIDENT + FIX — character ran into walls on Start (FIXED + deployed)
First live test of the two-field build: on Start the character started RUNNING into a
wall and Stop didn't halt it; owner powered off the VM.
- ROOT CAUSE: `_select_recipe` focused the search box with `ibgclick` (a separate
  scheduled task) and THEN fired a separate `run_ahk` to type. The window-focus gap let
  EQ2 lose field focus, so `type_field`'s `{Raw}` recipe name landed in the GAME WORLD —
  the w/a/s/d letters in "rawhide..." = movement keys. And `run_ahk` is fire-and-forget
  in the guest's interactive session, so `_stop` can't interrupt a type mid-flight =
  "Stop didn't stop it." This was a fail-OPEN violation of the chat/world-input invariant.
- FIX (deployed): (1) `guest.type_field(focus_xy=)` now does activate-EQ2 + click-field +
  type as ONE atomic AHK run (no foreground gap). (2) Dropped the trailing Enter (EQ2
  filters live; a stray world Enter opens chat). (3) New `sensors.search_box_text` /
  `search_landed` OCR the search field to PROVE the query landed; region derived from
  search_click/clear_click (override via `recipe_select.search_region`). (4) `_select_recipe`
  ABORTS the job on a focus miss instead of re-typing into the world; re-types only when
  focus is proven. Caps a focus failure to ONE short burst + clean stop, never wall-running.
- WATCH on next test: if the readback region is off, `search_landed` false-negatives and
  every craft ABORTS (safe, no movement) — recalibrate `recipe_select.search_region` from
  a screenshot if that happens.
