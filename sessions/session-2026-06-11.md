## Input injection — KEY FINDING (load-bearing)

  worked at first (login form) then stopped; `input-send-event` (QMP mouse) is
  accepted by QEMU but never moves the pointer (QXL/SPICE quirk; not the SPICE
  client — confirmed with no client + no vdagent). Do NOT rely on it.
- **In-guest AutoHotkey is THE injection method** — native Windows SendInput,
  100% reliable, and it's exactly what the bot agent will use at runtime.
  AHK v2 at `C:\ib\ahk\AutoHotkey64.exe`.
- **To launch a GUI/input process in the interactive session (1) from the agent

  PsExec -i 1 gave "Access denied"; schtasks /it `/run` gave "Element not found".
  The Register-ScheduledTask Interactive route works — used it to click "I ACCEPT".

## Display / resolution — KEY FINDING (deferred work)
- **RDP detaches the QXL console.** Win10 single-session: an RDP login moves
  iksar's session to `rdp-tcp` and leaves `console` (the QXL display the bot
  captures) inactive. This is what made QEMU input go flaky mid-session and what
  blocks console-side display changes. **Runtime fix (also used here):**
  `tscon <sessionid> /dest:console` redirects the session back to the physical
  console so capture/input keep working after an RDP disconnect. The bot must own
  the console; do not leave RDP connected while it runs.
- **Target resolution 1600x900** (16:9, just under 1080, DarkUI-friendly).
  QXL's `qxldod` exposes no settable 16:9 modes without spice-vdagent + a SPICE
  client (its native modes are 4:3). Confirmed: tasks run in the active console
  session (sid=2) yet EnumDisplaySettings returns zero modes; ChangeDisplaySettings
  fails. **Fix = switch VM video QXL -> virtio-gpu** (viogpudo driver already
  installed) which sets arbitrary resolutions headless. Needs a VM reboot, so
  DONE: switched video QXL->virtio-gpu and pinned 1600x900 via the EDID
  <resolution> element in the domain. Framebuffer now 1600x900. (Snapshot still
