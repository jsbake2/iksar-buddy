## In-guest loop: fixed the CLICK (the drag/cursor bug) — fast loop now reliable
The in-guest loop ran craft 1 but failed continuations: clicks landed as a DRAG and
grabbed the recipe icon onto the cursor (owner's "recipe on the pointer"). Root cause:
_click used SetCursorPos (a teleport) which emits NO WM_MOUSEMOVE, so EQ2 never
registered the cursor on the button → down/up misfired (on an icon = grab). Fix
(7478c20): _click now mirrors gclick_ev.ahk exactly — absolute mouse_event MOVE →
200ms settle → down(40ms)→up. After that the loop reliably completed 3/3 and 5/5
Carbonite Tower Shield IN-GUEST.

Start logic: BEGIN is the reliable start button (continuation prep window reappears
after each craft); the ↻ repeat click does NOT take. Proven version (1106ad9): try
begin (begin_lit), fall to repeat, confirm via running_detect (red stop-sign ~990px
running vs ~43 idle) with a 6s window, 4 attempts. Transition overhead ~10s/craft
(it tries ↻ first, ~6s, then begin) — still far better than the host path's ~34s.
A cadence-click rewrite (c9c5d34) to shave that ~6s FAILED at 1/4 (the continuation
craft starts ~9s AFTER the click, faster repeated clicks never wait through it) —
reverted to 1106ad9. guest_loop is ON; both crafter guests run the proven build.
