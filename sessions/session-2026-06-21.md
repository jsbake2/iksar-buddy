## Root cause
STOP was decoupled from the thing that presses keys. `controller.stop()` only called
`worker.stop()` (sets the host `_stop` flag) and set dashboard state‚Üíidle. The IN-GUEST
reflex (ib_agent ‚Üí craft_reflex.run_list) is what actually mashes arts in the VM, and it
was only told to idle *later*, when the host worker loop unwound and called
`_agent_set("idle")` from inside `_craft_list_via_agent`. So if the worker was mid
virsh-roundtrip / between handoffs at STOP time, the **dashboard read idle while the VM
kept crafting combines** ‚Äî exactly the owner's symptom.

Verified live: `POST /api/bot/B/stop` ‚Üí host `{ok:true}`, state idle, command channel
`{action:idle,epoch:58}`, guest status `state:idle` ‚Äî and a screenshot showed the craft
frozen mid-combine (55.3%, durability 100%, no further presses). So the host+guest stop
chain works once the guest receives idle; the gap was *latency/path* to the guest.
(Also confirmed the UI posts `/api/bot/${bot.id}/stop` with the correct A/B/C key ‚Äî not
the bug. My first manual curl used the dom name `iksar_buddy3` and got "unknown bot";
the keys are A/B/C.)

## Changed
- `forge/controller.py` `stop()` ‚Äî now calls `self.set_agent_command(bot_id, "idle")`
  directly, so the guest's next poll (~250ms) halts the reflex IMMEDIATELY instead of

- `forge/worker.py` `_craft_list_via_agent()` ‚Äî early `return 0` if `self._stop.is_set()`
