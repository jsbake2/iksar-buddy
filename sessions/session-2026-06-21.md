## Scope this session
Owner: "STOP DOES NOT FUCKING STOP IT" on a crafter. Make STOP actually halt the bot.

## Root cause
STOP was decoupled from the thing that presses keys. `controller.stop()` only called
`worker.stop()` (sets the host `_stop` flag) and set dashboard state→idle. The IN-GUEST
reflex (ib_agent → craft_reflex.run_list) is what actually mashes arts in the VM, and it
was only told to idle *later*, when the host worker loop unwound and called
`_agent_set("idle")` from inside `_craft_list_via_agent`. So if the worker was mid
virsh-roundtrip / between handoffs at STOP time, the **dashboard read idle while the VM
kept crafting combines** — exactly the owner's symptom.

Verified live: `POST /api/bot/B/stop` → host `{ok:true}`, state idle, command channel
`{action:idle,epoch:58}`, guest status `state:idle` — and a screenshot showed the craft
frozen mid-combine (55.3%, durability 100%, no further presses). So the host+guest stop
chain works once the guest receives idle; the gap was *latency/path* to the guest.
(Also confirmed the UI posts `/api/bot/${bot.id}/stop` with the correct A/B/C key — not
the bug. My first manual curl used the dom name `iksar_buddy3` and got "unknown bot";
the keys are A/B/C.)

## Changed
- `forge/controller.py` `stop()` — now calls `self.set_agent_command(bot_id, "idle")`
  directly, so the guest's next poll (~250ms) halts the reflex IMMEDIATELY instead of
  waiting for the host worker to unwind.
- `forge/worker.py` `_craft_list_via_agent()` — early `return 0` if `self._stop.is_set()`
  before the `_agent_set("craft_run")` handoff, closing the stop-then-handoff race (a
  stop landing during _select_recipe could otherwise (re)start the guest one more time).

## Decisions made
- Fix at the controller (immediate guest idle) rather than only tightening worker timing
  — it guarantees halt regardless of where the worker is in its state machine.
- Kept the worker's own `_agent_set("idle")` on unwind (belt-and-suspenders; both idle).

## Addendum — false DONE: gate craft completion on the EQ2 log, not pixels
Owner: crafts keep reading DONE when they're not (iksar_buddy2). Root cause confirmed:
in-guest `craft_reflex._react_until_done` declared done off PIXELS — after `active_grace`
with no counter it flipped `saw_active=True`, then a Create/Begin button on screen
(`_done()`) read as completion. But that button also shows BEFORE a craft starts, so a
craft that never really ran (or a table where counters mis-detect) false-completed
(`reactions=2` across "finished" crafts).

EQ2 log has the authoritative per-craft signal (verified on Cruelst's log):
  `You created \aITEM <id> <id>:<Item Name>\/a.`  — exactly one per finished craft
(success OR crit-fail). `Critical success/failure!` are per-counter, not whole-craft.

- `guest_agent/craft_reflex.py` — added `_find_log()` (newest eq2log_*.txt under the logs
  root = active char), `_log_size()`, `_new_creations()` (reads new bytes since a per-craft
  byte baseline, regex `_CREATED_RE`), `_craft_completed()` (any new creation = this craft
  done; bot crafts serially). `_react_until_done` now snapshots the log offset at craft
  start and DONE = a new created line. Kept a fast missing-mats bail (no counter + start
  button + grace + nothing logged). Pixel `_done` retained only as a fallback when no log
  is found. Removed the false `saw_active`-flip→pixel-done path.
- `forge/worker.py` — `_ruleset()` sends `done_via_log` (default true) + `eq2_log_root`;
  `_ruleset_craft_run(count, name)` sends `expected_item` (logged for visibility, not gated).
- Verified `_CREATED_RE` against the real log: matches all created lines, ignores
  Critical/`can only do this when crafting`.
