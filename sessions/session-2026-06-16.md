## Decisions made
- Recipe source = EQ2U/EQ2wire, not DBG Census. Confirmed (again) Census `recipe`
  collection has only name/bench/id — no level/tier/class, no recipebook collection.
  EQ2U has level+class but is behind login + Cloudflare → Playwright required.
- Scrape mechanism: Playwright logs in ONCE to clear Cloudflare, then bulk-fetch book
  pages via the authenticated request context (no per-page render). Server returns empty
  200s under sustained rate → hardened fetcher treats short/empty bodies as failures and
  RE-LOGINS (fresh ci_session + __cf_bm) + proactive re-login every 400 fetches. Raw HTML
  cached per book → resume-friendly.
- Classifier (book name → class): Advanced<Class>/archetype-essentials → class(es);
  artisan/crafter → generic "Artisan" (L1-9 + expansion all-class books the owner asked
  for); adorner/tinkerer in name → side classes; mastercrafted one-offs (Ancient
  Knowledge / Draconic Knowledge / Ancient Teachings) have no class in name → "Misc".
- Output layout (owner picked "Both"): canonical flat data (CSV + by_class JSON) stays
  denormalized for the bot; browse UI = class-first tier tree default + global-search flat
  table. UI deferred until data lands (schema is locked: class/level/category/book/recipe).
- Tinkerer/Adorner are skill-based (0-350 @ L70), not char-level → routed to `data/side/`
  keyed by book, not level. Precise 0-350 skill number is NOT on the recipe page text;
  capturing it needs a per-recipe detail pass (~20k fetches) → deferred as optional.
