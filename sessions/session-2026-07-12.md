# Session Log — 2026-07-12 (part 3): "buttons don't inject / spam to land" — key daemon down

## Symptom
Owner mid-raid: presses on the dashboard barely register in the VM ("spam the buttons").

## Diagnosis (live, 10.0.0.16 / VM iksar_buddy)
- Agent CONNECTED, chat_safe TRUE, not armed — but manual presses bypass arm, so that
  wasn't it. First real cause: **Asmoden's keymap was blank** → agent no-op'd every press
  (fixed: mirrored Joar's keys, see part 2 follow-up / commit d9a8da7-era).
- Deeper cause of the "spam": the **in-guest key daemon (ibkeyd) was dead** — its heartbeat
  (C:\ib\keydaemon.hb) frozen. With the daemon down, injects fall back to the slow one-shot
  path (~0.5s) and the `_injecting` mutex DROPS presses that arrive mid-inject → spam.
- Why the daemon couldn't self-heal: the VM rebooted ~05:53 PM today and wiped the ibkeyd
  task + `C:\ib\keyd.ahk`. The agent's self-heal wrote the script via the guest FILE API
  (`self.g.file_write` = guest-file-open), which on this guest can only OVERWRITE existing
  files, not CREATE new ones ("Access is denied" creating keyd.ahk; keycmd.txt overwrites
  fine). So once the script was gone it could never be rewritten → fast path stranded.
- NOT Windows Defender: RealTimeProtection=False, no CFA block events (1123/1124). The
  create failures look **load-driven** — the VM is CPU-pegged during the raid, so guest-exec
  (cold-starts powershell per op) times out / returns '' under load. Deploys succeeded in a
  combat lull (18:37) and failed under fire (18:48+). The daemon flaps healthy↔dead as the
  hb READ itself gets starved.

## Fixes (committed; deployed to ~/ib-app)
- `agent/host_agent.py`:
  - `_DAEMON_GUEST_SCRIPT`/XML → `C:\ib\ahk\keyd.ahk` (that subdir writes fine; C:\ib\keyd.ahk
    is create-denied). Daemon paths are absolute inside the script, so location is irrelevant.
  - `_IBKEYD_XML` Arguments → `C:\ib\ahk\keyd.ahk`.
  - `_deploy_daemon`: write via `self.g.push_bytes` (guest-exec `[IO.File]::WriteAllBytes`,
    CREATE-capable) instead of the file API; register with `schtasks /Create /XML` (reliable)
    instead of `Register-ScheduledTask -Xml` (returned '' → misread as failure).
  - Health check: ignore BLANK hb reads (guest-agent noise) and require TWO consecutive
    genuine stale reads before declaring the daemon dead — kills the flap that dropped us to
    the slow path on a single bad read.
- Manually restarted the daemon several times; it comes up in lulls, drops under load.

## Restart done (owner asked "restart it all, leave my VM running")
- `systemctl restart ib-brain ib-forge`, `start ib-agent`. VM iksar_buddy LEFT RUNNING.
  All three services active; agent reconnected.

## STILL OPEN / owner-facing
1. Key daemon stabilizes once the VM isn't CPU-saturated (raid over / lull). Under heavy
   combat the guest-agent channel is too slow to deploy/health-check reliably. Longer-term:
   give the VM more vCPU headroom, and/or make ibkeyd survive reboots (persist the task so a
   reboot doesn't wipe it). Meanwhile keys DO land via the one-shot fallback, just slower.
2. **PROFILE/CHAR MISMATCH**: dashboard active profile = `asmoden` (conjuror), but the VM is
   logged in as **Furyflatulence** (Fury healer — combat log shows Regrowth/Autumn's Kiss/
   Porcupine). So the pushed keymap (Asmoden, mirrored from Joar) does NOT match the in-game
   char's hotbars — presses fire the wrong abilities. Owner must pick: relog the VM to Asmoden
   OR set the profile back to furyflatulence.
