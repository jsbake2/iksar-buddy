## NEXT TASK (tomorrow's focus): make the bot a BETTER, FASTER healer
Owner: "it's slow and does not react well to my health getting low. It notices the
issue but lets me die too often." Reaction latency on the TANK is the problem.

Latency chain to attack (each adds delay before a heal lands):
1. **Sense lag** — HostSensor reads HP from a `virsh screenshot` at decision_hz/period
   (~2Hz, every 0.5s) + screenshot+crop+scan time. So HP is up to ~0.5-1s STALE.
2. **Decision throttle (likely the biggest tunable win)** — brain `server.py`:
   `GLOBAL_GCD_S=0.9` + per-action cooldowns (`direct_heal` 1.6, `critical_heal` 1.4,
   `emergency_heal` 1.0, `group_heal` 2.0). After any action it won't issue another
   for GLOBAL_GCD, and won't repeat the same heal until its cooldown. A tank diving
   fast can cross 85%→dead inside that window. EMERGENCY heal is a *different* action
   so it isn't blocked by a standard-heal cooldown — but it only triggers <25% (hp_emergency).
3. **Inject + cast** — guest-exec → AHK → EQ2, then the heal's cast time.

Ideas to try (in order):
- **Tank-priority fast lane**: sense the tank's HP more often / react with no GCD when
  the tank is below critical. Maybe a dedicated tank-HP fast read.
- **Raise thresholds / tier earlier**: hp_critical 0.50, hp_emergency 0.25 may be too
  low given the latency — by the time it sees <25% + casts, he's dead. Try emergency
  ~0.40, critical ~0.65 (owner-tunable via /api/tunables — add them to the dashboard).
- **Predictive**: react to HP *velocity* (big drop since last cycle → pre-emptive
  emergency heal) not just absolute thresholds.
- **Cut emergency latency**: emergency_heal should bypass GLOBAL_GCD entirely and fire
  the instant tank < emergency, even mid-other-cast (it's meant to cancel).
- **Faster sense**: bump decision_hz / sensor rate, or a lighter tank-only HP crop.
- The ward heartbeat (mitigation) helps survivability too — confirm it's actually firing.
